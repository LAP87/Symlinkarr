## Remaining Improvement Tracks

### P1 — Module Decomposition (not started)

These are large-file refactors, best done one file per session in a feature branch:

1. **db.rs (~4600 lines)** → `db/mod.rs` + `db/queries.rs` + `db/migrations.rs`
   - 68 public functions, 25 importing files
   - Natural split: schema/migration (lines 1–500), struct definitions (lines 1–340), queries (lines 500–2823), tests (lines 2823–4586)
   - 37 dependents import `db` — all need `use crate::db` updated if path changes

2. **cleanup_audit.rs (~4600 lines)** → audit logic + remediation logic
   - 5 importing files
   - Natural split: `cleanup_audit/mod.rs` + `cleanup_audit/remediation.rs`

3. **config.rs (~2900 lines)** → `config/mod.rs` + `config/validation.rs`
   - 37 importing files — widest fan-in in the project
   - Natural split: struct definitions + defaults (lines 1–1000), load/resolution (lines 1000–1700), validation (lines 1700–2300), tests (lines 2300–2923)

### P3 — Observability Gaps (not started)

4. **auto_acquire.rs** — zero skip-reason tracking. Acquisition failures are `Blocked`/`NoResult`/`Failed` but no granular skip reasons like `linker.rs` provides.
5. **matcher.rs** — no skip reasons at all. Matching failures just don't produce links. No way for an operator to answer "why didn't this match?"

### Not Recommended

- Security hardening beyond current state (already adequate for *arr-class)
- New feature breadth
- AI/fuzzy matching
- UI rewrite
- Test-coverage-as-a-project
